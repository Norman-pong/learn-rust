## §2 功能需求

### §2.1 练习重置机制（just reset）

**问题**：93 题答案已填入 `exercises/src/*.rs`，`// I AM NOT DONE` 注释虽在但代码已能通过。学习者打开文件即看到完整答案，"先错后改"的学习路径失效。

**需求**：

| # | 需求 | 验收标准 |
|---|------|---------|
| 2.1.1 | `just reset` 命令：一键将 exercises 恢复到"待做"状态 | reset 后 `cargo test -- --ignored` 全部 FAIL（≥ 70 题失败） |
| 2.1.2 | `just reset <chapter>` 支持单章重置 | 仅重置指定章节，其他章节保持当前状态 |
| 2.1.3 | reset 不影响 `solutions/` 目录 | `solutions/` 文件内容不变 |
| 2.1.4 | reset 后每题保留 `#[test] #[ignore]` + `// I AM NOT DONE` + 题目注释 | 格式与 Architecture §3.3 一致 |
| 2.1.5 | 桩代码设计：用 `todo!()` 占位需填写的部分，保留题目上下文 | `cargo check` 通过（可编译），`cargo test -- --ignored` FAIL（panic） |

**重置策略**：

从 `solutions/` 反向生成桩——提取每个 test 函数签名和 assert 语句，将实现逻辑替换为 `todo!()`：

```rust
// reset 前的 exercise（答案已填入）
#[test]
#[ignore]
fn variables1() {
    // I AM NOT DONE
    let mut x = 5;      // ← 答案
    x = 7;               // ← 答案
    assert_eq!(x, 7);
}

// reset 后的 exercise（待做桩）
#[test]
#[ignore]
fn variables1() {
    // I AM NOT DONE
    // 修复：x 默认不可变，需要加 mut 才能重新赋值
    todo!("声明可变变量 x=5，然后赋值为 7");
    // assert_eq!(x, 7);
}
```

### §2.2 内容代码块验证（just check-docs）

**问题**：`src/` 下 20 篇 mdbook 文章中的 ```` ```rust ```` 代码块从未编译验证。

**需求**：

| # | 需求 | 验收标准 |
|---|------|---------|
| 2.2.1 | `just check-docs` 命令：提取所有 mdbook 文章中的 Rust 代码块并编译检查 | 输出报告：文件名 + 代码块位置 + 编译结果 |
| 2.2.2 | 区分"完整可运行代码"和"示意片段" | 完整代码块必须编译通过；示意片段跳过 |
| 2.2.3 | 标记机制：用 ```` ```rust,should-compile ```` 标记需要验证的代码块 | 未标记的默认为示意片段，跳过 |
| 2.2.4 | 修复所有已发现的编译错误 | `just check-docs` 零 error |

**实现方案**：

用 Rust 脚本（放在 `scripts/check-docs.rs`）或 justfile recipe：

1. 遍历 `src/**/*.md`，提取 ```` ```rust,should-compile ```` 块
2. 每个块写入临时文件，用 `rustc --edition 2024 --crate-type lib` 编译
3. 收集错误，输出报告

### §2.3 CLI 工具实用化（projects/01-tools/）

**问题**：`hello-rust` 是玩具，无实际用途。

**需求**：

| # | 需求 | 验收标准 |
|---|------|---------|
| 2.3.1 | 从 Norman 工作流中选定一个替代目标脚本 | DEVLOG.md 记录：脚本名、功能、频率、痛点 |
| 2.3.2 | 用 Rust 实现等价功能 | `cargo run -- <args>` 输出与原脚本一致 |
| 2.3.3 | 使用 Rust 核心特性 | 至少用到：clap 参数解析 + Result/? 错误处理 + 文件或网络 IO |
| 2.3.4 | 单元测试 ≥ 3 个 | 核心逻辑有测试覆盖 |
| 2.3.5 | DEVLOG.md 记录 ≥ 3 个真实卡点 | 每个含：症状 / 编译错误 / 理解 / 修复 |
| 2.3.6 | 工具复杂度控制 | 单文件 < 300 行，dependencies ≤ 5 crate |
| 2.3.7 | 在生产中替代原脚本 ≥ 1 次 | 实际调用记录写入 DEVLOG |

**工具选型**（待 Norman 确定后填入）：

> 如果暂无法确定替代目标，可先实现一个通用工具作为练习：
> - `json-fmt`：格式化/校验 JSON（练 serde + 文件 IO）
> - `log-grep`：日志过滤/统计（练 regex + 迭代器）
> - `batch-rename`：批量重命名（练 clap + 路径操作）

### §2.4 编译 warning 清理

| # | 需求 | 验收标准 |
|---|------|---------|
| 2.4.1 | 清理 exercises 的所有编译 warning | `cargo test --no-run` 零 warning |
| 2.4.2 | 修复 mdbook 内部死链 | `mdbook build` 零 broken link 警告 |

## §3 非功能需求

| 约束 | 决策 |
|------|------|
| 新增依赖 | check-docs 脚本用纯 std（不引 crate）；CLI 工具允许 clap + serde |
